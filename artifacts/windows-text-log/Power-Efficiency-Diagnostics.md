# Power Efficiency Diagnostics Reports

## Forensic value
`powercfg /energy` and `powercfg /sleepstudy` produce HTML reports documenting system state at the moment of scan:

- **energy-report.html** — 60-second diagnostic scan: running processes, loaded drivers, USB devices, power-policy state, device inventory
- **sleepstudy-report.html** — sleep/wake analysis: wake sources, wake reasons, sleep-cycle history

Default location: `%SystemRoot%\System32\`. Files persist until manually deleted.

## Why this matters for DFIR
The reports ARE timestamped state snapshots. If an IT admin generated one during the incident window (for troubleshooting), the report captures running processes / loaded drivers / connected USB devices at that moment — data normally only available via live memory capture.

## Concept reference
- None direct — timestamped content snapshot.

## Triage
```cmd
dir /a /t:w %SystemRoot%\System32\energy-report.html
dir /a /t:w %SystemRoot%\System32\sleepstudy-report.html
```

Open the HTML in a browser — structured tables by section. Search for:
- **USB Devices** section — confirms devices plugged in at report time
- **Driver Requests** — driver inventory
- **Platform Power Management Capabilities** — system-identity + processor info
- **Device Power Requests** — connected devices detail

## Cross-reference
- **USBSTOR / USB-Enum** — permanent device-history (report = spot check)
- **Amcache-InventoryDriverBinary** — persistent driver inventory (report = snapshot)
- **Prefetch powercfg.exe-*.pf** — execution evidence for powercfg itself
- **Security-4688 / Sysmon-1** — powercfg.exe process-creation event (who generated the report)

## Practice hint
On a lab VM (elevated): `powercfg /energy` — wait 60 seconds. Open `%SystemRoot%\System32\energy-report.html` in a browser. Note the USB Devices section lists your connected devices with serial numbers + VID/PID. Scroll to Driver Requests — full driver inventory with paths. This snapshot IS the DFIR artifact.
