# Filesystem Volume Serial

## What it is
A 32-bit serial number generated by `format.exe` (or equivalent) when a filesystem is initialized. Stored in the filesystem's boot sector. It is the *filesystem-level* volume identifier, distinct from two other things it is often confused with:

- **VolumeGUID** — 128-bit, Mount-Manager-assigned on first mount, lives in registry
- **DeviceSerial** — the hardware serial reported by the USB descriptor or OS-synthesized

FilesystemVolumeSerial survives from format to format? **No** — it's regenerated each time a volume is formatted. So it's useful for proving "this particular formatting of this volume was accessed," but *not* for proving "this physical device was accessed" across reformats. (For the latter, pivot to **MBRDiskSignature** or **DeviceSerial** — both sit below the filesystem layer.)

## Per-filesystem offsets

| Filesystem | VSN offset | Width | Authoritative artifact |
|---|---|---|---|
| NTFS     | 0x48 | 8 bytes (shell artifacts record the low 4) | `Boot` ($Boot) |
| FAT32    | 0x43 | 4 bytes LE | `FAT32-Boot` |
| exFAT    | 0x64 | 4 bytes LE | `exFAT-Boot` |
| FAT12/16 | 0x27 | 4 bytes LE | (unwritten — `FAT16-Boot` ghost) |

## Forensic value
The primary volume-identity pivot across shell artifacts (LNK, shellbags, jump lists). When a user opens a file on E: drive, Explorer writes a LNK file in Recent\ that captures the target's volume serial at that moment. Re-plug the same physical device after reformatting — same DeviceSerial, different FilesystemVolumeSerial. The mismatch is itself a forensic signal.

## Encoding variations

| Artifact | Where |
|---|---|
| ShellLNK | `LinkInfo\VolumeIDAndLocalBasePath\DriveSerialNumber` (uint32 LE) |
| ShellBags | embedded in shell-item lists as drive-node serial |
| Jump Lists | inside shell-item data of referenced targets |
| Prefetch | volume references in path strings (when exe launched from non-C:) |
